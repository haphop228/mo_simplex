## ❌ Что **не** сделано (известные ограничения)

### 1. Свободные и отрицательные переменные
**Статус:** не реализовано.

В UI ([`ui/index.html`](../ui/index.html)) виден блок «Ограничения на переменные» с
$\pm\infty$ (на скриншоте от профессора), но в [`ui/app.js`](../ui/app.js)
эти данные не собираются и не передаются в API. Солвер жёстко предполагает $x \geq 0$.

**Что нужно:** реализовать редукции из таблицы 1.2 в [`task/algorith.md`](algorith.md):
- свободная $x_j$ → $x_j = x_j^+ - x_j^-$;
- $x_j \leq 0$ → $x_j' = -x_j$;
- $x_j \geq l_j,\ l_j \neq 0$ → $x_j' = x_j - l_j$;
- $x_j \leq u_j$ → дополнительная строка $x_j \leq u_j$.

Хранить «карту перевода» в [`LinearProblem`](../core/models.py) и применять
обратное преобразование при формировании финального $x^*$.

### 2. Прямой ввод задачи в канонической форме
**Статус:** не реализовано.

Сейчас система всегда добавляет балансовые/искусственные. Если пользователь уже
ввёл задачу в виде $Ax = b,\ x \geq 0$ с готовым единичным базисом, то лишние
искусственные обернутся холостым прогоном фазы I.

**Что нужно:**
- Чекбокс в UI «Задача в канонической форме».
- Авто-детектор столбцов-«ортов» (по одному на каждое равенство) — если найдены, использовать их как начальный базис без добавления искусственных.

### 3. UI границ переменных
**Статус:** виден на скриншоте, но **отсутствует** в репозитории
([`ui/app.js`](../ui/app.js), [`ui/index.html`](../ui/index.html) — без блока
границ).

**Что нужно:** добавить вёрстку и сборку `lower_bounds` / `upper_bounds` в
функции `solve()` ([`ui/app.js`](../ui/app.js:97)); расширить API
[`Api.solve()`](../main.py:14) для приёма этих массивов.

### 4. Документация двойственного решения с учётом инверсии знаков
**Статус:** работает, но требует пояснения в выводе.

При $b_i < 0$ строка инвертируется (`<=` ↔ `>=`), что меняет знак
соответствующего $u_i$. В тестовых данных профессора (Задача 1) `u = [-2, 0]`
вместо ожидаемого $(2, 0)$ — это проявление этой конвенции, тест
([`test_problem1_min_with_geq_constraints`](../tests/test_two_phase.py))
проверяет именно $|u_i|$ через критерий $z^* = 6$, не значение знака.

**Что нужно:** в [`core/exporter.py`](../core/exporter.py) добавить раздел
«Соответствие знаков двойственных переменных» в финальный шаг, где явно
выводится $u^*$ для исходной (не приведённой) задачи.

## 💡 Возможные улучшения (опциональные, помимо плана)

### Алгоритмические
1. **Big-M метод как альтернатива двухфазному** — для случаев, когда нужно избежать переключения целевой функции (например, для интерактивной демонстрации сходимости с одной целью).
2. **Антициклирование Бленда теперь стоит, но Lexicographic rule** даст более устойчивое поведение на сильно вырожденных задачах.
3. **Detect ill-conditioned pivots** (хотя при `Fraction` это не актуально, при будущем переходе на `float` критично).
4. **Параметрический симплекс / ранжирование двойственных переменных** — для чувствительности к правым частям $b$.

### UX/визуализация
5. **Графический вывод** многогранника допустимых решений (для $n=2$) с подсветкой текущей вершины $x_B$ и направления $z_0$ — мощный учебный инструмент.
6. **Подсветка ведущего элемента** $\pi = z_0[s_0]$ в матрице $B$ перед pivot-операцией.
7. **«Сравнить с Гомори / Большим M»** — кнопка для запуска альтернативного метода и сравнения числа итераций.
8. **История задач** (localStorage) — повторно загружать недавно решённые задачи.
9. **Экспорт в LaTeX** (помимо Markdown) — для прямой вставки в академические работы.
10. **Тёмная тема** для ночной работы.

### Кодовая база
11. **`numpy.array` с `Fraction`-объектами** для чистоты векторных операций (или собственный класс `Matrix`).
12. **CI:** добавить GitHub Actions для прогона `pytest` на push.
13. **Type-checking:** наладить `mypy --strict` (сейчас Pylance подсвечивает несколько Optional-warning'ов в [`core/exporter.py`](../core/exporter.py) — корректные по логике, но требуют точечных `assert`).
14. **Docstring везде** + `mkdocs` для авто-генерации документации API.
15. **Property-based тестирование** через `hypothesis`: генерировать случайные совместные задачи и сравнивать $f^*$ с эталоном из `scipy.optimize.linprog`.
