## ✅ Что реализовано дополнительно (ранее было в ❌)

### 1. Свободные и отрицательные переменные
**Статус:** ✅ реализовано.

В [`core/solver.py`](../core/solver.py) метод [`_apply_variable_reductions()`](../core/solver.py) реализует все редукции из таблицы 1.2 [`task/algorith.md`](algorith.md):
- свободная $x_j$ (lb=None, ub=None) → $x_j = x_j^+ - x_j^-$ (расщепление);
- $x_j \leq 0$ → $x_j' = -x_j$ (инверсия столбца);
- $x_j \geq l_j,\ l_j \neq 0$ → $x_j' = x_j - l_j$ (сдвиг, правая часть $b \leftarrow b - A_j l_j$);
- $x_j \leq u_j$ → дополнительная строка $x_j \leq u_j$;
- фиксированная $x_j = l_j = u_j$ → подстановка константы.

Карта перевода `_var_map` хранится в солвере. Метод [`recover_original_x()`](../core/solver.py) применяет обратные преобразования для восстановления исходных переменных $x^*$.

В [`LinearProblem`](../core/models.py) добавлены поля `lower_bounds` и `upper_bounds`.

### 2. Прямой ввод задачи в канонической форме
**Статус:** ✅ реализовано.

- Чекбокс «Задача в канонической форме» в [`ui/index.html`](../ui/index.html).
- При включении все ограничения фиксируются как `=`, поля границ блокируются.
- Авто-детектор столбцов-ортов в [`SimplexSolver._build_canonical_basis()`](../core/solver.py): для каждой строки ищется столбец с единственной `+1` и нулями в остальных строках. Найденные орты используются как начальный базис без добавления искусственных переменных. Для строк без орта добавляется искусственная переменная.
- Параметр `canonical_mode=True` передаётся из JS через [`Api.solve()`](../main.py).

### 3. UI границ переменных
**Статус:** ✅ реализовано.

- В [`ui/index.html`](../ui/index.html) добавлен блок «Границы переменных» с полями `от:` / `до:` для каждой переменной.
- В [`ui/app.js`](../ui/app.js) функция [`buildBoundsBlock()`](../ui/app.js) динамически строит блок; функция [`solve()`](../ui/app.js) собирает `lower_bounds` / `upper_bounds` и передаёт в API.
- Поддерживаются значения: числа, пустая строка / `inf` / `+inf` (→ $+\infty$), `-inf` (→ $-\infty$).
- Подсказка рядом с каждой переменной показывает тип: `x ≥ 0`, `свободная`, `a ≤ x ≤ b` и т.д.
- [`Api.solve()`](../main.py) принимает `lower_bounds` / `upper_bounds`, парсит их и передаёт в `LinearProblem`.

### 4. Документация двойственного решения с учётом инверсии знаков
**Статус:** ✅ реализовано.

- В [`SimplexStep`](../core/models.py) добавлены поля `u_0_original` и `row_inverted`.
- Метод [`_compute_u0_original()`](../core/solver.py) вычисляет $u^*$ для исходной (не приведённой) задачи: инвертирует знак $u_i$ для строк, умноженных на $-1$ при нормализации $b \geq 0$, и учитывает направление оптимизации (min → знак меняется).
- В [`core/exporter.py`](../core/exporter.py) метод [`_render_dual_solution()`](../core/exporter.py) добавляет в финальный шаг раздел «Двойственное решение $u^*$» с:
  - $u^*$ для приведённой задачи;
  - $u^*_{\text{orig}}$ для исходной задачи;
  - пояснением, какие строки были инвертированы и почему знак изменился.

## 💡 Возможные улучшения (опциональные, помимо плана)

### Алгоритмические
1. **Big-M метод как альтернатива двухфазному** — для случаев, когда нужно избежать переключения целевой функции (например, для интерактивной демонстрации сходимости с одной целью).
2. **Антициклирование Бленда теперь стоит, но Lexicographic rule** даст более устойчивое поведение на сильно вырожденных задачах.
3. **Detect ill-conditioned pivots** (хотя при `Fraction` это не актуально, при будущем переходе на `float` критично).
4. **Параметрический симплекс / ранжирование двойственных переменных** — для чувствительности к правым частям $b$.

6. **Подсветка ведущего элемента** $\pi = z_0[s_0]$ в матрице $B$ перед pivot-операцией.
